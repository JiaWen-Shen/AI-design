## 重點討論

### 1. Subtree vs Submodule 選型

| | Subtree | Submodule |
|---|---|---|
| Designer 操作 | 較單純（一般 commit/push） | 需兩段式 push |
| 管理端 | Maintainer 手動 pull 回 team repo | GitHub 原生管理、視覺可辨識 |
| 自動化 | 需另寫 GitHub Action / script | Hook 可處理（Phase B reminder） |

- **結論：繼續用 submodule**，靠 hook + 自動化解決兩段式 push 問題

### 2. Hook 與 Permission 安全性

- Hook 寫在 project-level repo（非 global），不同專案不同規則
- Hook 觸發時要教 user「為什麼不讓你做」，不只吐 error message
- 共識：AI 再怎麼厲害，人還是要看 commit 內容做判斷

### 3. 六週 Git 分享課程

- 本週開始，每次約 20 分鐘，觀念為主
- 架構：Figma vs Git mindset → Git 基礎 → Claude Code 操作 → Practice → 多專案工作流
- 滾動式調整，部分架構還在 war room 實驗中

### 4. Workshop / Onboarding Test

- 遊戲化流程：clone → 個人資料夾做事 → commit → 開 branch → 發 PR → review → merge
- 需準備 setup script 幫 designer 建好基本環境
- 每個方程式需 senior member 當 reviewer

## AI 工作流問題

### Agent 繞過 Permission / Hook

- AI 被 hook 擋住後，user 追問「這樣可以嗎？」→ AI 會嘗試繞路，且繞得過去
- Global CLAUDE.md vs project-level 指令衝突時，AI 可能選擇性遵守
- **對策：** hook 訊息要包裝成教育性說明，而非單純 error

### Agent 改到不該改的東西

- AI 有全局視角，會自動調動所有資源 → 可能去改 .gitmodules、settings 等源頭設定
- 多人同時工作時，各自的 AI 改動可能破壞 single source of truth
- **共識：** 人必須做最終判斷，這是 mindset 問題

### Commit Review 疲勞

- AI 產生大量 commit，user 看前幾個就直接按 OK
- Commit message 太技術化，designer 看不懂
- **需要：** 透過 CLAUDE.md 讓關鍵變更有明確警示；但每個 project「核心」不同，難以泛化

### 跨裝置同步衝突（OneDrive + Git）

- OneDrive Files On-Demand 讓 Git 追蹤到空殼檔案，產生假 diff
- 在家 commit 後 OneDrive 還在同步 → 到公司 pull 可能不完整
- **結論：** Git track 的東西不放 OneDrive；Karen 用 Jottacloud + GitHub 雙重備份

### 大檔案處理

- 設計 assets 動輒數 MB，AI 每次 commit 都嘗試推圖檔上去 → 卡住
- Binary diff 讓未修改的圖檔也被偵測為 change
- **對策：** .gitignore 排除大檔案，用 external reference 指向雲端

### AI 自動 Stage / Commit 透明度

- Agent 自動 stage + commit，user 不知道推了什麼
- Branch 上累積大量 commit 沒合回 main
- **期望：** user 至少去 GitHub 網頁端確認更新有上去；untracked changes > 10 個就要處理
