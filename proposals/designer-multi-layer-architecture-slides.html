設計師專案分層架構提案

Karen Shen | 2026-04-16

1

Problem

Designer 在公司工作時會接觸多個層級的規範：

  • Company Policy — 全公司共用的規則
  • Department Guidelines — 部門層級的設計規範
  • Project Rules — 專案特定的 workflow

痛點：目前沒有標準化的方式讓 Agent 在正確時機引用這些規範

2

核心問題

Project Repo

Designer 需要 貢獻內容
（push 設計成果、參與協作）

Company / Dept Repo

Designer 只需要 遵守規範
（read-only、不需要 push）

不同類型的 Repo，應該用不同的機制管理

3

Proposed Solution

Repo 類型 Designer 角色 建議機制
Project 需要貢獻（push） Git Submodule
Company / Dept 只需遵守（read-only） Local Cache + TTL New
Personal 完全自主 雲端硬碟 / Personal Repo

什麼是 TTL？

TTL = Time To Live（存活時間）

快取機制的設定：「這份資料可以用多久，過期就要重新拿」

設定 TTL = 24 小時

Session 1（Day 1 早上 9:00）
  └─ Cache 不存在 → fetch 規範 → 存到本地

Session 2（Day 1 下午 2:00）
  └─ Cache 未過期 → 直接用本地（不 fetch）

Session 3（Day 2 早上 10:00）
  └─ Cache 過期 → 重新 fetch → 更新 cache

為什麼選 Local Cache + TTL？

方案 確保 Agent 讀到 Designer 操作 Token 消耗
每次 WebFetch ✅ 零 高
Git Sparse Checkout ❓ 要主動讀 需 clone + pull 低
Skill 化 ❓ 要主動載入 需下載 skill 低
Local Cache + TTL ✅ Hook 強制 零 低
