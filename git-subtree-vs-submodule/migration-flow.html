Subtree → Submodule Migration Flow

One-time migration for Peter's existing subtree workspace

Parent repo: REI-Project Path: teams/HIE/ Old folder: rei-hie (subtree) New folder: rei-hie (submodule)

Before — Subtree

REI-Project/

    rei-hie/ ← subtree

❌ Content embedded in parent repo
No trace / separate history
Team needs manual pull

→

REI-Project/
  teams/HIE/
    rei-hie/ ← submodule
               ↕
    Peter's personal repo

✅ Independent repo, Claude Code managed
Hooks + CLAUDE.md active

0
Pre-flight Checks
  • 1
    Confirm all changes committed + pushed in Peter's personal repo
    Peter's personal repo is the source of truth — not the team repo's old snapshot
    git -C <peter-local-repo> status
    git -C <peter-local-repo> log origin/main..HEAD --oneline
  • 2
    Get Peter's personal repo URL and confirm it's accessible
    git ls-remote <peter-repo-url> HEAD
  • 3
    Confirm GitHub write access to REI-Project (needed for Phase B)
    gh auth status
⚠️ Do not proceed if Peter has uncommitted or unpushed work in his personal repo. His repo is the source — all latest changes must be pushed before migration starts.
1
Set Up New Submodule
  • 1
    Add Peter's existing repo as submodule at a new temp path ⚠️ New name — old rei-hie still exists
    Use a different name to allow side-by-side comparison before deleting old folder
    git submodule add <peter-repo-url> teams/HIE/rei-hie-new
  • 2
    Add workflow template files into Peter's submodule:
    CLAUDE.md  git-operation-guide.md  .claude/settings.json  .claude/hooks/phase-b-reminder.sh
  • 3
    Commit template files to Peter's personal repo Phase A
  • 4
    Update parent repo pointer Phase B
    git add teams/HIE/rei-hie-new .gitmodules
    git commit -m "chore: add rei-hie-new submodule for Peter"
    git push origin <parent-branch>
2
Content Migration
  • 1
    Compare content: rei-hie-new/ (Peter's latest) vs rei-hie/ (old Day-1 snapshot)
    rei-hie-new is expected to have more/newer files — verify nothing is accidentally missing
    diff -rq --exclude=".git" teams/HIE/rei-hie teams/HIE/rei-hie-new
  • 2
    Confirm with Peter that rei-hie-new/ has everything correct and complete
    Peter's repo is authoritative — differences from old snapshot are expected and normal
3
Cleanup — Delete Old Subtree + Rename
  • 1
    Remove old subtree folder rei-hie from parent repo
    git rm -r teams/HIE/rei-hie
    git commit -m "chore: remove old rei-hie subtree folder after migration"
    git push origin <parent-branch>
  • 2
    Rename submodule rei-hie-new → rei-hie
    git mv teams/HIE/rei-hie-new teams/HIE/rei-hie
    git config -f .gitmodules submodule.teams/HIE/rei-hie-new.path teams/HIE/rei-hie
    git submodule sync
    git commit -m "chore: rename submodule rei-hie-new to rei-hie"
    git push origin <parent-branch>
  • 3
    Verify .gitmodules shows correct final path and Peter's repo URL
⚠️ Only delete rei-hie after Peter confirms rei-hie-new is complete.
4
Verification
  • 1
    Fresh clone test to confirm submodule initializes correctly
    git clone --recurse-submodules <parent-repo-url> /tmp/test-clone
    Verify teams/HIE/rei-hie/ has Peter's files (not empty)
  • 2
    Verify Phase B reminder hook fires after a test push from inside rei-hie
    Hook file: .claude/hooks/phase-b-reminder.sh
  • 3
    Run onboarding for Peter — confirm CLAUDE.md config fields are populated
    Fields: designer_name, github_account, clone_path, my_submodule_folder
  • 4
    Notify other team members to re-pull with submodule update
    git pull origin <parent-branch>
    git submodule update --init teams/HIE/rei-hie
  • 5
    Delete test clone
    rm -rf /tmp/test-clone
Note: Phase A = commit to Peter's personal submodule repo  |  Phase B = update REI-Project parent repo pointer  |  Template source: cross-team-submod/template/
